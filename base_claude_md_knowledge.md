# base_claude_md_knowledge

## CLAUDE.md とは何か

CLAUDE.md は、Claude Code のセッション開始時に自動的に読み込まれるマークダウンファイル。プロジェクトのコーディング規約、アーキテクチャの決定事項、ビルドコマンド、ワークフローなどを記述しておくことで、毎回同じ説明を繰り返す必要がなくなる。

Claude Code はセッション間でメモリを持たないため、CLAUDE.md が唯一の「永続的な記憶」として機能する。内部的にはシステムプロンプトの後にユーザーメッセージとして注入される仕組みで、厳密な設定ファイルではなくコンテキストとして扱われる。

ファイル名は大文字小文字が区別される。必ず `CLAUDE.md`（CLAUDEが大文字、.mdが小文字）でなければならない。


## メモリの階層構造

Claude Code は複数の場所から CLAUDE.md を読み込み、それらをマージする。より具体的（ネストが深い）な指示が優先される。

### 読み込み順と優先度

1. マネージドポリシー（組織レベル） — 企業が管理する設定。除外不可
2. `~/.claude/CLAUDE.md` — グローバル設定。全プロジェクトに適用
3. プロジェクトルートの `CLAUDE.md` — チーム共有の規約やスタック情報
4. サブディレクトリの `CLAUDE.md` — モジュール固有の指示
5. `.claude/rules/*.md` — glob パターンで条件付き適用されるルールファイル
6. `CLAUDE.local.md` — 個人用のプロジェクト設定（.gitignore 推奨）

### 読み込みのタイミング

- 作業ディレクトリより上の階層にある CLAUDE.md → セッション起動時に即座に読み込み
- 作業ディレクトリより下の階層にある CLAUDE.md → そのディレクトリのファイルに触れたときに遅延読み込み（lazy loading）
- 兄弟ディレクトリの CLAUDE.md → 読み込まれない

たとえばモノレポで `frontend/` にいる場合、`backend/CLAUDE.md` は読み込まれない。

### ディレクトリ構成の全体像

```
your-project/
├── CLAUDE.md                  ← プロジェクトのメイン記憶
└── .claude/
    ├── settings.json          ← 権限、モデル選択、フック設定
    ├── settings.local.json    ← 個人用設定（gitignore対象）
    ├── rules/                 ← パス指定で適用されるモジュール式ルール
    │   ├── code-style.md
    │   └── testing.md
    ├── skills/                ← 再利用可能なワークフロー
    │   └── my-skill/
    │       └── SKILL.md
    ├── agents/                ← 専門サブエージェント
    │   └── code-reviewer.md
    └── .mcp.json              ← 外部ツール連携（Notion, GitHub 等）

グローバル（全プロジェクト共通）:
~/.claude/
├── CLAUDE.md                  ← 個人のグローバル設定
└── skills/                    ← 全プロジェクトで使えるスキル
```


## 何を書くべきか

### 書くべきもの

- プロジェクト概要 — 一行で何のプロジェクトか伝える。「Next.js + Stripe の EC アプリ」程度で十分
- テックスタック — 使用言語、フレームワーク、主要ライブラリ
- アーキテクチャ — ディレクトリ構成、モジュール間の関係、重要なファイルの所在
- コーディングルール — named export vs default export、関数型 vs OOP など具体的に。ただしインデント幅やセミコロンのようなフォーマットルールはリンターに任せ、CLAUDE.md には書かない（後述「ハーネスエンジニアリング」参照）
- コマンド — ビルド、テスト、リント、デプロイの具体的なコマンド
- 落とし穴や注意点 — プロジェクト固有の罠、deprecated なモジュール、特殊な回避策
- ドメイン用語 — プロジェクト固有の専門用語やその定義
- MCP サーバーの使い方 — 設定済みの MCP ツールの使用指示

### 書くべきでないもの

- コードスタイルの細かいルール — リンターやフォーマッターに任せるべき。LLM は遅くて高コスト
- 頻繁に変わる情報 — 週単位で更新が必要なものはプロンプトで伝える方が適切
- 機密情報 — CLAUDE.md は Git にコミットされることが多いため、秘密鍵やパスワードは絶対に書かない
- 曖昧な指示 — 「コード品質に注意して」のような測定不能な指示は効果が薄い


## 書き方のベストプラクティス

### サイズ

1ファイルあたり 200 行以下を目安にする。30〜100 行が理想的。長くなりすぎると指示の遵守率が下がる。コンテキストウィンドウを圧迫するため、トークンの無駄遣いにもなる。

### 構造

- マークダウンの見出しと箇条書きで整理する
- 関連する指示はセクションにまとめる
- 長い散文は避け、短く行動可能な指示を書く
- 箇条書き形式の指示は段落形式より遵守率が高いという報告がある

### 具体性

違反したかどうか判定できるレベルの具体性が必要。

- 良い例: 「2スペースインデントを使う」「コミット前に `npm test` を実行する」
- 悪い例: 「コードを適切にフォーマットする」「変更をテストする」

### 強調

絶対に守ってほしいルールには「IMPORTANT:」「NEVER」「ALWAYS」などの強調語を使える。ただし多用すると効果が薄れるため、本当に重要なものだけに限定する。


## @import によるモジュール化

CLAUDE.md が大きくなりすぎた場合、`@path/to/file` 構文で外部ファイルをインポートできる。

```markdown
See @README.md for project overview
See @docs/api-patterns.md for API conventions
See @package.json for available npm scripts
```

- インポートは再帰的に最大5階層まで可能
- 初めてプロジェクトで外部インポートが検出されると、承認ダイアログが表示される
- 拒否するとインポートは無効化される

この仕組みにより、CLAUDE.md 本体はスリムに保ちつつ、詳細なガイダンスは別ファイルに分離できる。フロントエンドチーム、セキュリティチームなど、それぞれが自分の担当ファイルを管理する運用も可能。


## .claude/rules/ によるモジュール式ルール

`.claude/rules/` ディレクトリに置いた `.md` ファイルは自動的に CLAUDE.md と同じ優先度で読み込まれる。インポート記述は不要。

```
.claude/
└── rules/
    ├── code-style.md    # 15行
    ├── testing.md       # 20行
    ├── git-workflow.md  # 10行
    └── security.md      # 12行
```

- Git にコミットしてチーム全体で共有するのが前提
- 関心の分離ができ、マージコンフリクトも減る
- Claude Code v2.0（2026年1月）以降で対応


## Auto Memory（自動メモリ）

Claude Code v2.1.59 以降で利用可能。Claude が作業中に自分でメモを書き、セッション間で知見を蓄積する仕組み。

- 保存場所: `~/.claude/projects/<project>/memory/`
- メインファイルは `MEMORY.md`（200行上限）
- ビルドコマンド、デバッグのノウハウ、アーキテクチャメモ、コードスタイルの好みなどを記録
- 毎セッション保存するわけではなく、将来のセッションで有用と判断した情報だけを保存

### 操作方法

- `/memory` コマンドで現在読み込まれているファイル一覧の確認、auto memory の ON/OFF 切り替えが可能
- 「pnpm を使って、npm は使わないで」のように明示的に記憶を依頼することもできる
- 不要になった記憶は「pnpm のルールを忘れて」で削除できる

CLAUDE.md（人間が書く指示）と Auto Memory（Claude 自身が書くメモ）は補完的に機能する。


## /init コマンドで始める

`/init` コマンドをプロジェクトディレクトリで実行すると、Claude がコードベースを分析してスターターの CLAUDE.md を自動生成する。

- package.json、既存のドキュメント、設定ファイル、コード構造を読み取って生成
- 既に CLAUDE.md がある場合は上書きせず改善を提案
- 生成されたファイルはあくまで出発点。そこから不要なものを削り、足りないものを追加する

「削るほうがゼロから書くより楽」というのが多くのユーザーの実感。


## よくある失敗パターン

1. 矛盾する指示 — あるセクションで「セミコロンを使う」、別のセクションで「セミコロン不要」と書く
2. 曖昧な段落 — 「コード品質に注意」のような測定基準のない指示
3. 重複 — CLAUDE.md と `.claude/rules/` で同じルールを繰り返す
4. ファイルが大きすぎる — 200行を超えると追跡精度が低下する
5. 構造がない — マークダウンセクションなしで全部がごちゃ混ぜ
6. リンターの仕事をさせる — コードフォーマットのルールは決定論的ツールに任せるべき

CLAUDE.md をテストするには、Claude Code に「CLAUDE.md の指示を5つのポイントで要約して」と聞く。明確に要約できなければ、指示が曖昧か矛盾しているサイン。


## メンテナンスの考え方

- CLAUDE.md は「生きたドキュメント」として扱う
- コードレビューで規約違反が見つかったら、CLAUDE.md に追記する
- 大きなリファクタ後は Claude 自身に「ドキュメントを更新して」と頼むのも有効
- 月1回程度の定期レビューで、古くなった指示を削除・統合する
- `#` コマンドでセッション中に試したルールが効果的だったら、CLAUDE.md に昇格させる

## claudeMdExcludes

大きなモノレポで他チームの CLAUDE.md が意図せず読み込まれる場合、`claudeMdExcludes` で除外パターンを設定できる。glob 構文で絶対パスに対してマッチング。ユーザー設定、プロジェクト設定、ローカル設定、マネージドポリシーの各レイヤーで設定可能で、配列は各レイヤー間でマージされる。ただしマネージドポリシーの CLAUDE.md は除外できない。


## AGENTS.md との関係

2025年中頃、Sourcegraph・OpenAI・Google・Cursor らの協力で AGENTS.md が策定された。Linux Foundation 傘下の Agentic AI Foundation が管理。「1つのファイルで、どの AI エージェントにも対応」というコンセプトで、Claude Code、Cursor、GitHub Copilot、Gemini CLI、Windsurf など多数のツールが対応している。

チームで複数の AI コーディングツールを使う場合、AGENTS.md に共通指示を書き、CLAUDE.md には Claude 固有の設定だけを書くという運用が現実的。


## グローバル CLAUDE.md の活用例

`~/.claude/CLAUDE.md` に書くのは、プロジェクトを問わず常に適用したい個人的なルール。

記載する内容の例:
- 好みの言語やフレームワーク
- コミュニケーションスタイル（簡潔に、コード例を先に見せる、等）
- エラーハンドリングの方針（明示的な try-catch、意味のあるエラーメッセージ）
- コメントの方針（WHY を書く、WHAT は書かない）
- diff は最小限にする、既存コードを勝手に大幅改変しない、といった行動規範

ここに書くべきでないのは、特定プロジェクトにしか当てはまらない情報。グローバル設定は「どのプロジェクトでも成立する契約」として書く。


## ハーネスエンジニアリング: CLAUDE.md の外に出すべきもの

OpenAI のハーネスエンジニアリング論文（2026年2月）の核心は「環境設計 > モデル能力」。CLAUDE.md にすべてを詰め込むのではなく、決定論的に処理できるものはフックやリンターに委譲し、CLAUDE.md は軽量なエントリーポイントに留める。

### PostToolUse フック: コードフォーマットの自動化

Claude がファイルを編集するたびに Biome（フォーマッター）と Oxlint（リンター）を自動で走らせる。残った違反だけがエージェントのコンテキストに注入される。これにより CLAUDE.md にインデント幅やセミコロンのルールを書く必要がなくなる。HumanLayer の分析によれば「LLM にリンターの仕事をさせるな」が鉄則。LLM は遅くて高コストで、決定論的ツールの方が確実。

### PreToolUse フック: 設定ファイルの保護

biome.json、tsconfig.json、next.config などのリンター/ビルド設定ファイルへの編集を、exit code 2 で物理的にブロックする。CLAUDE.md に「設定ファイルを編集するな」と書くだけでは破られることがあるが、フックは Claude の思考や文脈に関係なくシェルスクリプトレベルで強制される。

### プリコミットフック: 最終防衛線

lefthook 等でコミット前に typecheck、lint、format-check を並列実行する。Claude が `git commit --no-verify` を使わない限り（CLAUDE.md で禁止する）、品質チェックをすり抜けることはない。

### 複利効果

エージェントが新しい種類のミスをしたとき、それを防ぐテストまたはリンタールールを追加する。一度追加したルールはすべての将来のセッションに適用される。これがハーネスの複利効果。

### 指示の配置先マッピング

| 内容 | 配置先 | 理由 |
|------|--------|------|
| コマンド、アーキテクチャ概要、完了の定義 | CLAUDE.md | 毎セッション必要 |
| セキュアコーディングルール | .claude/rules/security.md | 長いため分離 |
| コードフォーマット | PostToolUse フック | 決定論的に処理 |
| 設定ファイル保護 | PreToolUse フック | 物理的にブロック |
| コミット前検証 | lefthook.yml | Claude の判断に依存しない |
| 機密ファイル除外 | .claudeignore | コンテキストに入れない |
| 権限制御 | settings.json | deny + フックの多層防御 |


## 知識管理: 地図を渡せ、マニュアルではなく

OpenAI チームが最初に学んだ教訓は「1つの大きな AGENTS.md に全部書く」アプローチの失敗。コンテキストウィンドウは有限なので、巨大な指示ファイルがタスクやコードを押し出してしまう。

解決策は CLAUDE.md を「目次」（約100行）にして、構造化された `docs/` を正式な記録システムにすること。

```
CLAUDE.md             ← 目次
docs/
├── design-docs/      ← 設計ドキュメント
├── exec-plans/       ← 実行計画（アクティブ/完了/技術的負債）
│   ├── active/
│   └── completed/
└── adr/              ← アーキテクチャ決定記録
```

「段階的開示（progressive disclosure）」のパターンで、エージェントは短い CLAUDE.md から始まり、必要に応じてより深いドキュメントをたどる。CLAUDE.md に進捗メモや設計判断の本文を直接書くと、蓄積するにつれて古い情報が新しい情報を押し出し、結局「最新のしか読まない」状態になる。

### ローカル CLAUDE.md: リスク箇所への道標

サブディレクトリに小さな CLAUDE.md を置くと、Claude がそのディレクトリのファイルに触れたときだけ遅延読み込み（lazy loading）でコンテキストに入る。プロジェクトルートの CLAUDE.md のトークンを消費せず、関係ない作業のときはノイズにならない。

認証・決済・マイグレーションなど、ミスの影響が大きい領域に配置する。

```
app/api/auth/CLAUDE.md     — セッション管理の方針、トークン検証手順
app/api/billing/CLAUDE.md  — 金額計算の精度、二重課金防止パターン
lib/db/CLAUDE.md           — マイグレーション変更禁止、クエリのパラメータ化必須
```

最初のセットアップで空のファイルを用意する必要はない。リスク箇所が明確になったタイミングで追加していく。プロジェクトルートの CLAUDE.md のアーキテクチャセクションに「リスクの高いディレクトリには個別の CLAUDE.md を配置する」という方針だけ書いておけば十分。

### docs/ の具体化

`docs/adr/` だけでなく、プロジェクトの規模に応じて以下も検討する。

- `docs/architecture.md` — システム全体のアーキテクチャ概要。CLAUDE.md から `See @docs/architecture.md` で参照
- `docs/runbooks/` — 運用ランブック。デプロイ手順、障害対応手順など
- `docs/design-docs/` — 設計ドキュメント。機能の設計経緯と判断理由

いずれも段階的に必要になったタイミングで追加する。小さなプロジェクトで最初から全部用意するのはオーバーエンジニアリング。


## セッション間の作業引き継ぎ: GitHub Issue の活用

Claude Code はセッション間でメモリを持たない。Auto Memory は補助的だが、作業の進捗状態を確実に引き継ぐには外部ストレージが必要。GitHub Issue に Handover ドキュメントを残す運用が有効。

### CLAUDE.md に書く内容

```markdown
## 作業引き継ぎ

PR 作成時またはセッション終了時に、GitHub Issue で引き継ぎ記録を残す。

- `gh issue create -l handover` で作成する
- タイトル: `[Handover] {作業内容の要約}`
- 本文に含めること:
  - 完了した内容（箇条書き、各項目1行）
  - 未完了の内容（箇条書き）
  - 次のセッションへの申し送り（判断が必要な点やブロッカー）
  - 関連する PR 番号
- 本文に含めないこと: 実装の経緯、試行錯誤の記録、コードの説明
- 設計判断は docs/adr/ に記録する（Issue には書かない）

新しいセッション開始時は `gh issue list -l handover` で直近の Issue を確認してから作業を始める。
```

### 粒度の指針

- トリガーは「PR 作成時」と「セッション終了時」の2つ。小さな変更ごとには書かない
- 1つの Issue は読んで30秒で状況が把握できる長さに留める
- 「次のセッションの Claude が何も聞かずに作業を再開できるか」が判断基準
- `handover` ラベルを付けることで通常の Issue と混ざらない

### 情報の種類ごとの置き場所

| 情報の性質 | 置き場所 | 理由 |
|---|---|---|
| 作業の進捗・引き継ぎ | GitHub Issue | 流動的。セッションごとに変わる。外部なので CLAUDE.md を圧迫しない |
| 設計判断・仕様・方針 | docs/ 以下の構造化ドキュメント | 半永続的。Git 管理できる。段階的開示で必要時だけ参照 |
| 上記への「道案内」 | CLAUDE.md | 目次。「Issue を読め」「docs/ を参照しろ」という数行の指示だけ |

### 別のアプローチ: /handover コマンドと HANDOVER.md

セッション終了時に `/handover` カスタムコマンドを実行し、プロジェクトルートに HANDOVER.md を生成させるアプローチもある。今回のセッションでやったこと、判断の理由、次やるべきことなどを Claude にまとめさせる。次のセッション開始時にこのファイルを読み込ませることで文脈を復元する。

GitHub Issue 方式と比較した場合の特徴:

- ファイルシステム上にあるので Claude が参照しやすい（gh コマンド不要）
- GitHub を使っていない環境（ローカルのみの個人開発、GitLab 等）でも使える
- セッションが非常に長く複雑で、Issue の箇条書きでは状況が伝わらないケースに向く

一方で注意点もある:

- 中身が肥大化しやすい。「バグの修正履歴」「設計判断の理由」「学んだ教訓」まで含めると長くなり、次のセッションで読まれなくなる
- セッションが重なると上書きか複数ファイルの散乱が起きる。Issue なら時系列で自然に並ぶ
- pre compact hook での自動化は形式的な内容になりがちで品質が下がるという報告がある。手動発火の方が品質が保たれる

GitHub Issue 方式と併用する場合は、HANDOVER.md を `.gitignore` に入れてローカル専用の個人メモとして使い、チームの正式な引き継ぎ記録は GitHub Issue に残すのが現実的。CLAUDE.md に入れて強制する必要はなく、必要な人が個人の運用として採用すれば十分。


## Codex との併用

Claude Code と OpenAI の Codex CLI を同一端末にインストールすると、`codex exec` コマンドで相互に呼び出せる。実装計画のレビューとコードレビューを Codex に委任するワークフローが有効。

### 計画レビュー

実装計画をユーザーに提示する前に Codex でレビューする。Codex は瑣末な指摘をしがちなので「致命的な点だけ指摘して」と明示する。修正後は `resume --last` で文脈を引き継いで再レビュー。

### コードレビュー

実装完了後、コミットハッシュを Codex に渡してレビュー。判断基準はセキュリティ・ロジックの問題は必ず修正、スタイル・命名の指摘は無視（リンターの仕事）。

### CLAUDE.md への組み込み方

Codex レビューの手順は「毎セッション必要なワークフロー」なので CLAUDE.md に含める。コマンド例とプレースホルダを書いておけば、Claude が毎回正しい手順で実行する。


## /simplify の位置づけ

Claude Code のビルトインスキル `/simplify` は、3つの並列サブエージェント（Code Reuse, Code Quality, Efficiency）でコードをリファクタリングする。重複の共通化、型の厳格化、アルゴリズムの改善に効果的。

ただし毎回の実装完了後に走らせるべきではない。理由は3つ:

- テスト前にリファクタリングすると、リファクタリングの正しさを検証するテストが存在しない状態で構造を変えることになる（TDD の RED → GREEN → REFACTOR の順序に反する）
- 3つのサブエージェント並列起動のコスト（トークン消費と所要時間）が小さな変更に対して割に合わない
- PostToolUse フックや Codex レビューとの役割重複

適切な使いどころは、機能の実装が一段落してテストが通った後のリファクタリングフェーズ、PR 作成前の最終チェック、技術的負債が溜まったときのスポット適用。CLAUDE.md の完了の定義には入れず、必要なときに手動で実行する。


## セキュリティ: 3層の防御構造

Claude Code の環境セキュリティは .claudeignore、settings.json の deny、PreToolUse フックの3層で構成する。

### なぜ settings.json の deny だけでは不十分か

- deny ルールが正しく機能しないバグが複数報告されている（GitHub Issue #6699, #8961 等）
- `Read(.env)` を deny しても `grep "" .env` は Bash ツール経由なので Read の deny が効かない
- `python3 *` や `node *` を allow に入れると urllib や fetch() で curl/wget の deny を迂回できる

### .claudeignore

最上流の対策。指定したファイルはコンテキストに入らないため、grep・cat・awk 経由の間接読み取りを含めてゼロにできる。

### セキュアコーディングルール

Claude が生成するコードの脆弱性対策。CLAUDE.md 本体に書くと長くなりすぎるため `.claude/rules/security.md` に分離する。入力バリデーション（Zod）、SQL インジェクション、XSS、コマンドインジェクション、プロトタイプ汚染、JWT、シークレット管理などを DO/DON'T 形式で記述。


## 遵守率を上げるテクニック: `<important if>` タグ

HumanLayer の Dex 氏が提案した手法。Claude Code は CLAUDE.md を `<system_reminder>` タグで包んで「関連があるかもしれないし、ないかもしれない」という前置きでモデルに渡すため、ファイルが長くなるほど個々のセクションが「任意」として扱われがちになる。

`<important if="condition">` タグでセクションを囲むと、いつ適用すべきかの信号が明確になり遵守率が上がる。

```markdown
<important if="you are writing or modifying tests">
- Use `createTestApp()` helper for integration tests
- Mock database with `dbMock` from `packages/db/test`
- Test fixtures live in `__fixtures__/` directories
</important>
```

ただしこれはワークアラウンドであり、先にやるべきことがある:

1. CLAUDE.md を短くする（200行以下）
2. リンターの仕事をリンターに任せる（PostToolUse フック）
3. 長いルールは `.claude/rules/` や `@import` で分離する

これらの根本対策をやった上でなお遵守率が低いセクションがある場合に `<important if>` を試す。すべてのセクションを囲むと効果が薄れるため、テスト固有の規約やデプロイ手順など、特定条件でのみ適用すべきルールに限定する。


## コンテキスト管理

コンテキストウィンドウは Claude Code の最重要リソース。ファイルの読み込み・コマンド出力・会話履歴がすべて積み上がり、満杯に近づくとパフォーマンスが劣化する（初期の指示を「忘れる」、ミスが増える）。

### 基本ルール

- 無関係なタスクを切り替えるときは `/clear` でコンテキストをリセットする
- 同じ問題で2回以上修正が発生したらコンテキストが汚染されているサイン。`/clear` して、学んだことを踏まえたより具体的なプロンプトで再開する（長いセッションに累積した失敗アプローチが判断を妨げている）
- `Esc` でいつでも中断できる。コンテキストは保持されるので修正指示を出してそのまま続けられる
- `Esc + Esc` または `/rewind` で会話とコードの状態を任意の時点に巻き戻せる

### コンテキストを節約するテクニック

- 外部サービスとのやり取りは CLI ツール（`gh`, `aws`, `gcloud` 等）を使わせる。API レスポンスをそのまま貼るより大幅にトークンを節約できる
- 大きなファイルを丸ごと読ませず、必要なセクションだけ指定する
- ログや diff の出力は `head -50` 等で先頭のみ渡す
- CLAUDE.md を短く保つ（200行以下）ことがコンテキスト節約の根本対策

### セッション運用のパターン

| 状況 | 対処 |
|---|---|
| 別タスクに切り替える | `/clear` |
| 同じ問題で2回以上修正した | `/clear` + より具体的なプロンプトで再開 |
| 間違った方向に進んでいる | `Esc` で中断 → 修正指示 |
| 試した変更を取り消したい | `/rewind` で巻き戻し |
| セッションをまたいで再開したい | `/resume` でセッション選択 |


## まとめ

- CLAUDE.md は Claude Code における最高レバレッジのファイル。30分の投資で毎週何時間も節約できる
- 簡潔・具体的・構造化が3原則。200行以下、箇条書き、違反判定可能な粒度
- `/init` で始めて、削って、育てる
- 大きくなったら `@import` や `.claude/rules/` で分割
- Auto Memory との併用で、人間が書く指示と Claude が学ぶ知見を補完的に運用
- Git にコミットしてチームで共有。CLAUDE.local.md で個人差分を管理
- AGENTS.md との組み合わせでマルチツール環境にも対応
- コードフォーマットはフックに、セキュリティルールは .claude/rules/ に、進捗は GitHub Issue に。CLAUDE.md は「目次」に徹する
- Codex との併用で計画レビューとコードレビューを自動化。手順は CLAUDE.md に含める
- `/simplify` は都度ではなく、テスト通過後のリファクタリングフェーズでスポット実行
- セキュリティは .claudeignore + settings.json deny + フックの3層防御。deny 単独には頼らない
- 遵守率が低い場合は `<important if>` タグを試せるが、まず CLAUDE.md を短くすることが先
- コンテキストは最重要リソース。同じ問題で2回以上修正が出たら `/clear` して再開する。無関係なタスクの切り替えにも `/clear` を使う
